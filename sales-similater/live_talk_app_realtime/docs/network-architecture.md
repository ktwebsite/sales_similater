# Cloud Run における固定IP化 (Cloud NAT構成) について

## 概要
本プロジェクトでは、Cloud Run からの外部通信（Outbound通信）において、**固定IPアドレス**を使用する構成を採用しています。
Kintoneなどの「IPアドレス制限」がある外部サービスと連携するために必須の構成です。

## 1. なぜこの構成が必要なのか？

### 通常のCloud Runの挙動
Cloud Run はサーバーレスサービスであり、インスタンスが起動するたびに Google が保有する膨大なIPアドレスプールからランダムなIPが割り当てられます。
そのため、**IPアドレスは不定（毎回変わる）** です。

### 直面した課題
連携先の **Kintone** はセキュリティポリシーとして「IPアドレス制限」を行っており、許可されたIP以外からのAPIアクセスを遮断（401/403エラー）します。
Cloud RunのIPが毎回変わるため、Kintone側に「このIPを許可してほしい」と申請することができませんでした。

## 2. 解決策：Cloud NAT アーキテクチャ

この問題を解決するために、以下のGCPリソースを組み合わせた構成をとっています。

### 通信フロー
`[Cloud Run]` → `[VPCコネクタ]` → `[Cloud NAT (固定IP)]` → `[インターネット]` → `[Kintone]`

1.  **VPCコネクタ (Serverless VPC Access):**
    Cloud Run と VPCネットワークを繋ぐブリッジです。Cloud Runからの通信を強制的にVPC内部へ引き込みます。
2.  **Cloud NAT:**
    VPCからインターネットに出る通信の「送信元IP」を、予約済みの**静的IPアドレス**に書き換えます。
3.  **固定IPアドレス:**
    Kintone側には、このCloud NATに紐づいたIPアドレスのみを許可リストに登録してもらいます。

## 3. この構成を採用すべきケース
今後、以下の要件が発生した場合も同様の構成（または既存リソースの使い回し）が必要です。

* **IP制限のあるSaaSとの連携:** Kintone, Salesforce, 銀行系APIなど。
* **オンプレミスDBへの接続:** 社内ネットワークやVPN越しのDB接続が必要な場合。
* **セキュリティ要件:** 「通信元を特定できる必要がある」と監査等で指摘された場合。
